# Wings Sidecar 迁移方案 v5（解耦契约补充版）

更新时间戳：`20260226-180748`

## 1. 修订背景

基于 v4/v4-codex，本版新增两项关键补充：

1. 明确“解耦后不再在拼命令阶段 `source` vLLM 虚拟环境”。
2. 补齐“控制层与引擎层”的交互契约（共享卷协议 + 状态机 + 时序）。

当前仍为方案设计阶段，不改代码。

---

## 2. 范围与边界（MVP）

- 单机、非分布式。
- 仅 vLLM。
- 端口固定：
  - 引擎后端：`17000`
  - 对外业务：`18000`
  - 健康探针：`19000`
- 继续优先复用 `wings/wings/proxy`（业务代理 + health 逻辑）。

---

## 3. 架构分层（解耦后）

## 3.1 控制层（wings-infer）

职责仅包括：

- 参数解析与配置合并（对齐 `wings_start.sh` 语义）。
- 引擎命令拼装（产物写入共享卷）。
- 启动/监管 proxy（18000）与 health（19000）。
- 生命周期与信号处理。

不负责：

- 在本容器内直接拉起引擎进程。
- 管理引擎运行时虚拟环境（venv/conda）。

## 3.2 引擎执行层（开源组件执行器）

职责包括：

- 监听共享卷命令产物。
- 按产物启动 vLLM 并监听 `17000`。
- 回写进程状态、PID、错误信息到共享卷。

## 3.3 对外服务层（复用）

- `proxy` 提供业务接口（18000）。
- `health_service + health.py` 提供健康接口（19000），内部探测 `127.0.0.1:17000/health`。

---

## 4. 命令拼装约束（新增）

## 4.1 禁止项

控制层生成的命令中，禁止出现以下运行时耦合操作：

- `source <venv>/bin/activate`
- `conda activate ...`
- 硬编码宿主机 Python 解释器路径（如 `/opt/xxx/venv/bin/python`）

## 4.2 允许项

- 仅输出“可执行命令 + 参数 + 环境变量键值”。
- 运行时环境由引擎执行层保证（镜像内已具备 `vllm` 或对应 Python 环境）。
- 推荐优先输出参数化结构（JSON），shell 脚本仅作为兼容产物。

---

## 5. 控制层-引擎层交互协议（新增核心）

## 5.1 共享卷目录约定

约定路径（示例）：

- `/shared-volume/engine/start_command.json`
- `/shared-volume/engine/start_command.sh`（可选兼容）
- `/shared-volume/engine/runtime.env`（可选）
- `/shared-volume/engine/engine.pid`
- `/shared-volume/engine/status.json`
- `/shared-volume/engine/last_error.log`

## 5.2 `start_command.json` 契约

最小字段建议：

- `request_id`：本次启动请求唯一标识
- `created_at`：生成时间
- `engine`：固定 `vllm`
- `backend_port`：固定 `17000`
- `argv`：数组形式命令（避免转义歧义）
- `env`：环境变量键值对（不包含激活 venv 的命令）
- `work_dir`
- `health_url`：`http://127.0.0.1:17000/health`
- `startup_timeout_sec`
- `log_path`

## 5.3 状态文件 `status.json` 契约

建议状态：

- `generated`：控制层已生成命令
- `accepted`：执行层已接收
- `starting`：引擎拉起中
- `running`：进程已存活
- `ready`：`17000/health` 就绪
- `failed`：启动失败
- `stopped`：已退出

最小字段建议：

- `phase`
- `updated_at`
- `pid`（若有）
- `reason`（失败原因，可空）
- `exit_code`（若有）

## 5.4 原子写入规则

控制层写共享卷时采用：

1. 先写临时文件（如 `start_command.json.tmp`）。
2. `fsync` 后原子 `rename` 为正式文件名。
3. 最后更新 `status.json` 为 `generated`。

避免执行层读取到半文件。

---

## 6. 启动时序（v5）

```text
container entrypoint
  -> python -m app.main                         # 控制层
       -> 参数解析/配置合并
       -> 生成 start_command.json(.sh) 写入共享卷（原子）
       -> 启动 proxy 进程（18000）
       -> 启动 health 进程（19000，复用 wings/wings/proxy）
       -> 前台守护 + 信号清理

engine executor（独立组件）
  -> 监听共享卷命令
  -> 拉起 vLLM（17000）
  -> 回写 status.json / pid / error

health 进程（19000）
  -> 周期探测 127.0.0.1:17000/health
  -> 输出阶段性健康状态
```

说明：

- 控制层不阻塞等待 `17000` 再启动自身服务。
- readiness 语义由 `19000` 健康服务统一对外表达。

---

## 7. 健康与探针策略

- 保持 `19000` 为唯一探针入口。
- 推荐 readiness 使用 `exec` 方式解析响应体（如 `p=="ready"` 且 `backend_ok==true`），避免 `httpGet` 将冷启动 `201` 误判成功。
- liveness 可继续 `httpGet /health`（19000）。

---

## 8. 示例产物（无虚拟环境激活）

`start_command.sh`（兼容）示例：

```bash
#!/usr/bin/env bash
set -euo pipefail
exec vllm serve /weights \
  --host 0.0.0.0 \
  --port 17000 \
  --served-model-name ${MODEL_NAME}
```

`start_command.json`（推荐）示例：

```json
{
  "request_id": "req-20260226-001",
  "created_at": "2026-02-26T18:07:48+08:00",
  "engine": "vllm",
  "backend_port": 17000,
  "argv": ["vllm", "serve", "/weights", "--host", "0.0.0.0", "--port", "17000"],
  "env": {"MODEL_NAME": "qwen2.5-7b-instruct"},
  "work_dir": "/workspace",
  "health_url": "http://127.0.0.1:17000/health",
  "startup_timeout_sec": 600,
  "log_path": "/shared-volume/engine/engine.log"
}
```

## 9. 最小可验证流程（MVP）

1. 控制层启动后，确认共享卷出现 `start_command.json` 且字段完整。
2. 执行层接收命令后，`status.json` 从 `generated -> accepted -> starting -> running/ready` 迁移。
3. `127.0.0.1:17000/health` ready 后，`127.0.0.1:19000/health` 进入 ready 语义。
4. 业务流量通过 `18000` 正常转发到 `17000`。
5. 发送 `SIGTERM` 后，控制层与子进程清理完整，状态回写为 `stopped` 或带退出原因。

---

## 10. 风险与对策（v5新增关注）

风险 1：执行层实现不统一，导致共享卷协议漂移。  
对策：冻结 `start_command.json/status.json` 字段并加版本号（如 `schema_version`）。

风险 2：仍残留 `source venv` 依赖，导致跨镜像不可移植。  
对策：控制层产物校验加入“禁止关键字检查”（`source`/`conda activate`）。

风险 3：命令转义问题导致执行失败。  
对策：以 `argv` 数组为主，`.sh` 仅兼容；失败信息统一回写 `last_error.log`。

风险 4：readiness 提前放行。  
对策：沿用 v4-codex 建议，readiness 使用 `exec` 对响应体字段判定。

---

## 11. 结论

v5 相比 v4 的核心增量是“明确解耦契约”：

- 控制层只做决策、拼命令、服务编排，不承担虚拟环境激活与引擎进程直接拉起。
- 引擎执行层通过共享卷消费命令并回写状态，形成可观测闭环。
- `wings/wings/proxy` 继续作为 18000/19000 的复用基础，保持改造面最小化。
