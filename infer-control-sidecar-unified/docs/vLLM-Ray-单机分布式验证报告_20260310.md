## 四、关键设计决策与问题解决

### 4.1 hostNetwork 方案失败 → Pod 网络方案

| 阶段 | 方案 | 结果 |
|------|------|------|
| 初始方案 | `hostNetwork: true` | ❌ 两个 Pod 共享宿主 IP (7.6.16.150)，`_determine_role()` 无法区分 Master/Worker |
| 最终方案 | **Pod 网络 (CNI)** | ✅ 每个 Pod 获得独立 IP (10.42.1.x / 10.42.1.y) |

**根本原因**: 系统设计依赖每个 Pod 有独立的 `RANK_IP` 来构建节点间通信。`hostNetwork` 下同机两个 Pod 的 `status.podIP` 相同，导致 `_determine_role()` 中 `MASTER_IP == RANK_IP` 对两个 Pod 都为 True。

**CNI 验证结果**:
```
test-cni-0   10.42.1.2   Running   5b590a43256c
test-cni-1   10.42.1.3   Running   5b590a43256c
```
同节点两个 Pod 分配了不同 IP，满足分布式要求。

### 4.2 CoreDNS 跨节点不可达 → hostPath IP 交换

**问题**: 两个 k3s 节点（Docker-in-Docker）的 `INTERNAL-IP` 都是 `172.17.0.3`（Docker bridge 默认分配），导致 flannel 无法在 Server 节点 (148) 和 Agent 节点 (150) 之间路由流量。CoreDNS Pod 运行在 Server 节点 (10.42.0.x)，Agent 节点上的 Pod (10.42.1.x) 无法访问。

**解决方案**: hostPath IP 交换机制
```yaml
volumes:
- name: ip-exchange
  hostPath:
    path: /tmp/wings-ip-exchange
    type: DirectoryOrCreate
```

```bash
# wings-infer 启动脚本中的 IP 交换逻辑:
export NODE_RANK=${POD_NAME##*-}
echo $POD_IP > /ip-exchange/pod-${NODE_RANK}-ip

if [ "$NODE_RANK" = "0" ]; then
  export MASTER_IP=$POD_IP
  # 等待 worker 写入 IP
  for i in $(seq 1 60); do [ -f /ip-exchange/pod-1-ip ] && break; sleep 2; done
  WORKER_IP=$(cat /ip-exchange/pod-1-ip)
  export NODE_IPS="${POD_IP},${WORKER_IP}"
else
  # 等待 master 写入 IP
  for i in $(seq 1 60); do [ -f /ip-exchange/pod-0-ip ] && break; sleep 2; done
  export MASTER_IP=$(cat /ip-exchange/pod-0-ip)
  export NODE_IPS="${MASTER_IP},${POD_IP}"
fi
```

**效果**: Pod 启动后 2 秒内完成 IP 交换，无需依赖 DNS 解析。

### 4.3 NODE_RANK 角色判断

`_determine_role()` 原逻辑依赖 `MASTER_IP == RANK_IP` 判断角色。在 IP 交换方案下，增加 `NODE_RANK` 环境变量作为更直接的判断依据：

```python
# main.py _determine_role()
node_rank_env = os.getenv("NODE_RANK", "").strip()
if node_rank_env:
    role = "master" if node_rank_env == "0" else "worker"
```

`NODE_RANK` 从 StatefulSet Pod 名称的序号派生 (`infer-0` → `0`, `infer-1` → `1`)。

### 4.4 DNS 解析修复 (_wait_and_distribute_to_workers)

**问题**: `NODE_IPS` 包含的是 Pod IP (如 `10.42.1.8,10.42.1.9`)，而 Worker 注册时也上报 Pod IP。但若 `NODE_IPS` 含有 DNS 名称（如 `infer-1.infer-hl`），需要解析后再比较。

**修复**: 在 `_wait_and_distribute_to_workers()` 中增加 `_resolve()` 函数：

```python
def _resolve(host: str) -> str:
    """Resolve DNS name to IP; return as-is if already an IP or lookup fails."""
    try:
        return socket.gethostbyname(host)
    except socket.error:
        return host

# 使用:
resolved_workers = {_resolve(ip) for ip in worker_ips}
if resolved_workers.issubset(registered):
    ...
```

### 4.5 Worker Health 端口偏移

同机两个 Pod 有独立 IP（Pod 网络），端口原则上不冲突。但为保险起见，Worker 的 health 端口偏移 +1：

| Pod | Health 端口 |
|-----|------------|
| infer-0 (Master) | 19000 |
| infer-1 (Worker) | 19001 |

### B. 排障技巧

1. **多层 Shell 引号问题**: PowerShell → SSH → docker exec → crictl exec → sh，推理测试 curl 命令的引号被逐层剥离。解决方案：将 curl 命令 base64 编码后写入 hostPath 共享卷，在 Pod 内 `base64 -d` 解码后执行。

2. **Docker BuildKit 缓存导致镜像不更新**: `DOCKER_BUILDKIT=1` 模式下即使源文件已修改，`COPY` 层可能命中缓存。解决方案：使用 `DOCKER_BUILDKIT=0 docker build --no-cache`。

3. **k3s-in-Docker INTERNAL-IP 冲突**: 两个 k3s 节点通过 Docker bridge 启动时，`INTERNAL-IP` 都是 `172.17.0.3`，导致 flannel VXLAN 无法建立跨节点隧道。对于同节点 Pod（本次场景），不受影响；跨节点需使用 `--node-external-ip` 或 `--flannel-iface` 指定宿主机真实 IP。
